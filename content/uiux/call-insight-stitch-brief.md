# 1. 产品简介

## 1.1 产品名称

**Call Insight｜全周期拜访与客户洞察平台**

## 1.2 一句话定位

帮助一线业务人员完成访前准备、访后复盘和行动跟进，并通过结构化数据与具体客户声音，为管理者提供基础业务洞察。

## 1.3 要解决的问题

一线业务人员在拜访前，需要分散查找客户历史、上次遗留事项、产品资料和竞品资料；拜访后，客户反馈、观念变化和后续行动又常被记录在不同工具或个人笔记中。

管理者通常只能看到拜访数量，难以快速理解：

- 客户具体关注什么；
- 对竞品的反馈如何；
- 客户观念是否变化；
- 哪些事项仍未闭环；
- 团队需要什么支持。

Call Insight 将一次拜访的准备、复盘、跟进和洞察串联为统一的移动端流程。

## 1.4 MVP 边界

- 移动端优先。
- 使用预置 Mock Data。
- 业务数据通过 localStorage 本地保存。
- 不接 AI API、CRM、真实数据库和真实权限系统。
- 产品知识库只读。
- 业务模拟数据可以重置。
- 视角切换仅用于 Demo，不代表真实身份认证。

# 2. 一线视角与管理视角

## 2.1 一线视角

**主要用户：** 一线业务人员 / CFC

### 可以做什么

- 新建和编辑自己的拜访；
- 查看同客户历史和未完成行动项；
- 完成访前准备；
- 选择聚焦产品、多个预设竞品和对比维度；
- 查看独立竞品对比卡；
- 生成访前准备卡；
- 完成访后复盘；
- 添加 1–3 条关键客户反馈；
- 创建和更新行动项；
- 查看个人洞察。

### 不能做什么

- 查看其他一线人员的明细；
- 修改产品与竞品知识库；
- 新增系统竞品；
- 生成 AI 洞察、治疗建议或外部推广话术。

## 2.2 管理视角

**主要用户：** DM、LMM、MKT

### 可以做什么

- 查看全部 Mock 数据的汇总；
- 查看核心数字和重点洞察；
- 查看客户声音；
- 查看行动项、到期情况和支持需求；
- 下钻到反馈详情、关联拜访和对比快照。

### 不能做什么

- 新建拜访；
- 修改一线拜访记录；
- 替一线人员关闭行动项；
- 评价个人绩效或生成拜访质量评分。

> DM、LMM、MKT 在本 MVP 中共用同一套管理视角，不设计三个独立版本。

# 3. 核心业务流程

## 3.1 一线完整闭环

**新建拜访 → 客户与目标准备 → 选择产品 / 竞品 / 维度 → 查看竞品对比卡 → 生成访前准备卡 → 完成拜访 → 访后复盘 → 记录反馈 → 创建行动项 → 形成洞察**

## 3.2 新建拜访五步向导

1. 客户与拜访信息；
2. 目标与客户观念；
3. 产品、竞品与对比维度；
4. 独立竞品对比卡；
5. 访前准备卡。

第三步完成后，**必须先进入第四步竞品对比卡，不能直接跳到访前准备卡。**

## 3.3 管理流程

**查看概览 → 筛选时间 / 产品 / 主题 → 阅读重点洞察 → 下钻客户声音 → 查看关联拜访、行动项与支持需求**

## 3.4 拜访状态流转

**草稿 → 待拜访 → 待复盘 → 跟进中 / 已完成**

| 状态 | 含义 | 编辑规则 |
|---|---|---|
| 草稿 | 访前信息尚未提交 | 访前内容可编辑，可删除 |
| 待拜访 | 已完成访前准备 | 访前内容可编辑，可修改日期 |
| 待复盘 | 已完成拜访，尚未复盘 | 访前只读，可填写复盘 |
| 跟进中 | 已复盘，存在未完成行动项 | 复盘和行动项可修改 |
| 已完成 | 已复盘且无需跟进，或行动项全部完成 | 默认只读，可主动重新编辑复盘 |

# 5.5 F04｜新建拜访 Step 1：客户与拜访信息

## 进度展示

顶部显示“1 / 5”，并用清晰步骤名表示当前阶段。

## 字段

- 客户姓名：自由输入，必填，最多 20 个字符；
- 医院：预设选择，必填，不支持新增；
- 科室：预设选择，必填；
- 职称：根据科室动态过滤，必填；
- 拜访日期：默认当前日期，可修改；
- 拜访方式：线下拜访 / 线上拜访。

## 科室—职称联动

### 临床科室

ICU、呼吸科、血液科、感染科、普外科、急诊科、移植科、儿科、烧伤科、老年科。

可选职称：

- 主任医师；
- 副主任医师；
- 主治医师；
- 住院医师。

### 检验科

可选职称：

- 主任技师；
- 主管技师；
- 技师。

### 药剂科

可选职称：

- 主任药师；
- 副主任药师；
- 药师。

切换科室后，如果原职称不适用，必须清空并要求重新选择。

## 客户历史匹配

按“客户姓名 + 医院 + 科室”匹配。

匹配成功后，页面显示一张历史卡：

- 上次拜访日期；
- 上次访后观念；
- 总体总结；
- 未完成行动项；
- 上次遗留事项。

未匹配时显示：

> 暂无该客户历史拜访记录，本次将作为首次拜访。

# 5.6 F05｜新建拜访 Step 2：目标与客户观念

## 主要拜访目标

采用“预设标签 + 具体目标自由输入”。

PRD 允许的目标标签：

- 探寻客户观念；
- 传递核心信息；
- 回应客户顾虑；
- 产品与竞品沟通；
- 跟进医学问题；
- 跟进使用反馈；
- 其他。

## 客户观念阶段

1. 尚未形成认知；
2. 已有基础认知；
3. 认可部分价值；
4. 愿意尝试或深入了解；
5. 已有使用及认可。

## 字段

- 当前观念阶段；
- 当前观念说明；
- 目标观念阶段；
- 目标观念描述；
- 待探寻问题；
- 上次遗留事项。

## UI 要点

- 观念阶段可以使用纵向单选卡或简洁阶梯；
- 不要设计成评分仪表；
- 不要暗示对员工或客户进行价值评分。

# 5.7 F06｜新建拜访 Step 3：产品 / 竞品 / 维度

## 选择规则

- 单次只选择 1 个聚焦产品；
- 至少选择 1 个预设竞品；
- 竞品数量不设上限；
- 多个竞品保留选择顺序；
- 至少选择 1 个对比维度；
- 维度数量不设上限；
- 建议优先选择 4 个以内，以提高移动端阅读效率；
- 一线用户不能新增竞品；
- 不做智能推荐。

## 聚焦产品

- 原研头孢哌酮/舒巴坦（舒普深）；
- 艾沙康唑（康新博）；
- 原研头孢他啶/阿维巴坦（思福妥）；
- 氨曲南/阿维巴坦（思福诺）；
- 伏立康唑（威凡）；
- 舒巴坦/度洛巴坦（鼎优乐 / Xacduro）。

## 对比维度

### 临床与产品

- 目标病原菌或耐药机制；
- 适用感染类型；
- 临床疗效证据；
- 安全性；
- 特殊人群使用；
- 给药方案；
- 肾功能调整；
- 指南或共识证据。

### 支付与管理

- 单价；
- 日治疗费用；
- 医保报销条件；
- DDD 或相关管理口径。

## UI 要点

- 产品使用单选卡；
- 竞品使用可排序的多选列表；
- 已选竞品在顶部形成横向标签；
- 维度使用分组多选；
- 底部主按钮：“查看竞品对比”；
- 未达到“至少 1 个竞品 + 1 个维度”时，按钮禁用。

# 5.8 F07｜新建拜访 Step 4：竞品对比卡

这是访前准备的核心工作页，不能被简化成一句摘要。

## 顶部上下文

- 聚焦产品；
- 已选竞品数量；
- 已选竞品列表；
- 已选维度；
- “返回修改”入口。

## 固定信息条

每个产品显示：

- 对比展示名称；
- 商品名 / 品牌；
- 规范通用名 / 成分；
- 规格 / 剂型；
- 给药途径。

## 多竞品移动端布局

- **固定我方产品列**；
- 竞品列横向滑动；

# 6. 不能丢失的关键业务规则

1. **PRD 是页面和规则的最高优先级。** Excel 中出现与 PRD 文案不一致时，以 PRD 为准。
2. 新建拜访必须是五步流程，中途退出自动保存草稿。
3. 第三步后必须进入独立竞品对比卡，不能直接生成准备卡。
4. 单次只选择 1 个聚焦产品。
5. 至少选择 1 个竞品和 1 个维度。
6. 竞品和维度数量均不设上限，且保留用户选择顺序。
7. 多竞品使用“固定我方列 + 竞品横向滑动列”。
8. 不能因为移动端空间有限而删除已选竞品或维度。
9. 访前准备卡必须完整保留所有已选竞品、维度、顺序、结论、来源和状态。
10. 标记已拜访后，comparison_snapshot 冻结并只读。
11. 历史拜访继续显示当时的数据版本和查询日期，知识库更新不能覆盖历史快照。
12. 客户历史按“客户姓名 + 医院 + 科室”匹配。
13. 科室和职称必须联动。
14. 一次拜访只能添加 1–3 条关键客户反馈。
15. 客户反馈原话和一线人员补充判断必须视觉区分。
16. 客户声音列表不展示客户姓名，详情页才展示虚拟姓名。
17. 有未完成行动项时拜访为“跟进中”，行动项全部完成后自动变为“已完成”。
18. 管理视角不能修改一线记录或关闭行动项。
19. 管理筛选只有时间、产品和反馈主题，不增加角色、团队、地区或医院筛选。
20. 产品资料显示来源、更新时间 / 查询日期、适用口径、确认状态和必要备注。
21. 关系必须通过 Relationship_ID 匹配，不得通过名称猜测。
22. 商品名 / 品牌只作补充展示，不作为产品—竞品关系主键。
23. 价格、日治疗费用、医保和 DDD 必须显示具体 SKU、计算口径、置信度和风险提示。
24. DDD 只是管理口径，不等同于推荐剂量。
25. 缺失或不确定数据只能显示“暂无可靠数据 / 待业务确认 / 不适用”，不得猜测补齐。
26. 统计结果必须从明细数据重新计算，不能只读取 Excel 的 Dashboard。
27. “重置演示数据”只重置业务数据，不重置只读产品知识库。
28. 不支持跨设备同步。

### 关系 1：艾沙康唑 vs 伏立康唑

- Relationship_ID：R-ISA-001；
- 临床场景：侵袭性曲霉病初始治疗；
- 直接证据：有直接 RCT；
- 关键研究：SECURE（T001）；
- 研究设计：随机、双盲、活性对照非劣效；
- 主要终点：42 日全因死亡率；
- 结构化结果：艾沙康唑达到非劣效；总体药物相关不良事件更少；
- 局限性：不可外推为所有真菌或所有患者的全面优效；
- 对比结论：可陈述非劣效和安全谱差异，不扩大为全面优效；
- 风险标签：direct_trial、qt_direction、interaction；
- 来源：SRC-IDSA-ASPER-2016、SRC-TRIAL-SECURE；
- UI 数据状态：公开来源已核对；
- 固定提示：公开来源已核对不等同于正式推广口径。

### 关系 2：艾沙康唑 vs 泊沙康唑

- Relationship_ID：R-ISA-002；
- 临床场景：侵袭性霉菌病治疗 / 预防；
- 直接证据：无直接 RCT；
- 疗效证据摘要：证据来自不同场景，不做数值横比；
- 对比结论：先区分治疗 / 预防、曲霉 / 毛霉和剂型；

### 关系 3：艾沙康唑 vs 伊曲康唑

# 9. 移动端 UI 方向

## 9.1 整体定位

- 移动端优先；
- 浅色背景；
- 蓝绿色医疗专业风格；
- 清晰、克制、可信；
- 适合企业内部业务工具；
- 不是消费级社交产品；
- 不是市场推广落地页；
- 不是高饱和科技大屏。

## 9.2 建议视觉语言

- 页面背景：极浅灰蓝或浅灰绿；
- 主卡片：白色；
- 主色：深蓝绿 / 医疗青绿；
- 辅助色：低饱和蓝；
- 风险和逾期：克制的橙红；
- 成功和已完成：中性绿色；
- 边框：浅灰蓝；
- 阴影：非常轻；
- 圆角：中等，不做过度圆润；
- 图标：简洁线性图标；
- 插画：少量、抽象，不使用真实患者或医生照片。

## 9.3 信息层级

- 顶部标题清楚；
- 主任务按钮始终明显；
- 重要状态用标签区分；
- 复杂数据使用“首层核心信息 + 展开详情”；
- 不把核心结论藏在弹窗或二级 Tab；
- 长文本采用分段、引用卡和可展开区域；
- 来源和风险提示使用小字号但保持可读。

## 9.4 导航

### 一线底部导航

- 首页；
- 拜访；
- 洞察；
- 我的。

### 管理底部导航

- 概览；
- 客户声音；
- 行动与支持；
- 我的。

底部导航固定，当前项明确。

## 9.5 五步向导

- 顶部显示步骤数字和步骤名称；
- 底部固定“上一步 / 下一步”；
- 保存草稿状态轻提示；
- 长页面保留底部操作；
- 必填错误在字段下显示，并自动滚动到首个错误。

## 9.6 竞品对比页

这是整套 UI 中最重要、最需要突出差异化的页面：

- 固定我方列；
- 竞品横向滑动；
- 顶部显示“已选 3 个竞品”；
- 每个维度独立成块；
- 维度标题固定醒目；
- 对比结论使用克制的结论条；
- 数据状态使用标签；
- 来源、局限和风险提示可展开；
- 不使用复杂雷达图代替文字证据；
- 不把多个竞品压缩成难以阅读的小表格。

## 9.7 管理概览

建议使用：

- 2 × 2 或横向滑动 KPI 卡；
- 横向条形图展示主题分布；
- 重点洞察主题卡；
- 客户原话引用卡；
- 行动状态摘要；
- 清晰的近 7 / 30 / 90 天切换。

避免：

- 过多仪表盘；
- 3D 图表；
- 过度渐变；
- 小字号密集表格；
- 以图表替代客户具体声音。

## 9.8 企业工具细节

- 最小点击区域约 44px；
- 正文保持舒适行距；
- 重要按钮使用清晰动词；
- 禁用状态明显；
- 所有状态标签保持同一语义颜色；
- 客户原话使用引用样式；
- 数据状态与业务状态使用不同视觉系统；
- 页面顶部或“我的”中持续说明虚拟数据和内部使用边界。

# 10. 禁止事项

1. 不新增 PRD 中没有的功能。
2. 不生成 AI 自动摘要、自动洞察、竞品推荐或下一步话术。
3. 不生成新的医学结论、治疗建议、患者用药推荐或个体化治疗方案。
4. 不生成未经审核的外部推广话术或传播内容。
5. 不评价员工绩效，不生成个人排名或拜访质量评分。
6. 不展示医院准入、进药、采购状态或医院级价格匹配。
7. 不实现真实登录、真实权限、跨角色派单、消息提醒、审批流或多人协作。
8. 不接 CRM、真实数据库或真实客户数据。
9. 不使用真实患者信息。
10. 不允许一线用户新增竞品。
11. 不把商品名或品牌作为关系主键。
12. 不按名称猜测 Relationship_ID。
13. 不根据零散字段自行推导产品优劣结论。
14. 不自动抓取或编造价格、医保、DDD 和日治疗费用。
15. 不把 DDD 当作推荐剂量。
16. 不删除用户已经选择的竞品或维度。
17. 不用隐藏 Tab 代替多竞品横向滑动。
18. 不把全部对比结论合并成一个模糊的总摘要。
19. 不在管理客户声音列表直接展示客户姓名。
20. 不将“我的访前备注”纳入管理洞察。
21. 不将预设关键词称为 AI 洞察。
22. 不在缺失或歧义时猜测补齐数据。
23. 不用高饱和、娱乐化或营销感强的 UI。
24. 不用真实医生、患者或医院照片作为主要视觉。
