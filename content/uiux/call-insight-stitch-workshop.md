# Part B｜Call Insight × Stitch：从线框图到可交付 UI

## 这一课要完成什么

基于 Call Insight 的低保真线框图和设计简报，使用 Stitch 生成第一版高保真移动端 UI，再把结果交接给开发 Agent，最终得到可运行的前端 Demo。

## 开始前准备

准备三份材料：低保真线框图、完整 Stitch 设计简报，以及 Part A 中确认的页面资产台账和设计规则。它们的优先级是：**业务规则与流程 > 页面结构 > 视觉风格 > 局部文案**。

不要只上传一句“帮我设计一个客户拜访工具”。AI 需要知道角色差异、五步向导、竞品对比、隐私边界和管理指标，才能画出正确的产品，而不是一张营销海报。

## 第 1 步：生成第一版高保真 UI

把线框图和设计简报一起交给 Stitch。首轮只要求它先完成入口、一线首页、新建拜访向导、访后复盘和管理概览等代表性页面，并保留移动端的真实内容密度。

检查首稿时，先看流程和层级，再看漂亮与否：

- 五步向导是否能持续展示当前进度。
- 一线人员与管理者是否使用不同的信息重点。
- 竞品对比是否能容纳多个竞品和多个维度。
- 管理指标是否有明确的数据口径。

## 第 2 步：用问题导向的提示词优化

不要笼统地说“再优化一下”。针对问题选择明确的修改指令：

1. 统一整体风格：统一卡片、标题、按钮与状态颜色。
2. 页面太拥挤：减少同屏字段，把低频信息收入展开区。
3. 页面像营销海报：降低装饰，增加任务、数据和操作优先级。
4. 优化线框图布局：保持原有的信息顺序和页面关系。
5. 区分一线和管理视角：一线聚焦下一步行动，管理聚焦聚合趋势。
6. 保留多竞品对比：禁止把复杂对比压缩成单一结论卡。
7. 优化访前准备卡：输出可执行的对话目标、注意事项和证据来源。
8. 简化管理概览：先呈现核心指标，再逐层进入主题和明细。

每次只修改一个问题，并在修改后回到线框图复核，避免“越改越偏”。

## 第 3 步：查看原型与 AI Studio

高保真 UI 图、即时原型和 AI Studio 中的可运行页面是三种不同产物：UI 图用于确认视觉与布局；原型用于体验跳转与状态；AI Studio/开发 Agent 用于把明确设计落成代码。不要把一张静态截图当作已经完成了交互设计。

## 第 4 步：把设计交接给 Agent

通过 MCP 或项目文件把 Stitch 页面交给 Agent。先验证 Agent 能看到项目和页面，再要求它根据页面生成 `DESIGN.md`。这份文档应沉淀颜色、字体、间距、组件、导航、状态、交互和移动端规则，成为后续开发的一致依据。

## 第 5 步：生成移动端 Demo

交给开发 Agent 时，明确材料优先级与实现范围：先实现主路径与关键状态，不要一开始追求所有页面。至少覆盖：入口、一线首页、新建拜访 5 步、访前准备、访后复盘、管理概览和必要的空/错状态。

快速检查时，问 Agent：页面是否遵循 DESIGN.md？流程是否按简报顺序？是否出现了无来源的数据、越权功能或虚构的业务规则？

## 第 6 步：形成更新闭环

Stitch 修改某一页面后，不要只截图发给开发。同步更新对应页面、DESIGN.md 和验收清单，说明“什么变了、为什么变、影响哪些状态”。这样下一轮生成才不会覆盖已确认的设计决策。

## 本节完成标准

- 第一版 UI 覆盖了核心页面和主流程。
- 每次优化都能对应到一个具体问题和一条规则。
- Agent 已成功读取页面，并生成或更新 DESIGN.md。
- 前端 Demo 能走通主路径，且没有越过业务和隐私边界。
