# Call Insight：用 AI 工具组合完成 H5 UI 设计与开发交付操作指南

## 目标

将 PRD 从产品理解、低保真流程、核心页面风格探索、全量 UI 补齐、全局统一，到开发交付，转化为可检查、可复用的协作流程。

适用对象：医疗行业 H5 工具、同时包含一线代表端与管理端的产品。

## 本文中的工具称谓

本文不绑定某一种 AI 工具组合。可按团队现有工具替换：

| 本文称谓 | 可选工具示例 | 常见用途 |
|---|---|---|
| 对话式 AI | ChatGPT、Gemini | 阅读 PRD、梳理流程、生成低保真概念、讨论需求与文案。 |
| 设计/开发 Agent | Codex、Antigravity | 组织页面资产台账、生成/修改 UI、制作交互画板、输出开发交付物。 |

同一个任务也可由同一工具完成；重点是保留每一步的输入、输出和验收，而不是固定使用某个平台。

---

## 开始前准备

在任何 UI 设计前，至少准备以下资料：

1. 最新 PRD：产品目标、用户角色、业务流程、功能范围、数据边界。
2. 低保真流程图：包含主要页面、步骤及跳转关系。
3. 视觉参考图：H5 尺寸、顶部风格、图标风格、卡片风格、列表密度等。
4. 内容确认表：每个重要页面的目的、入口、主要操作、字段分组、默认示例数据、特殊状态与出口。

其中第 4 项不要求一开始就把全部文案定稿，但必须先确定页面的信息结构和主要操作。

---

## 推荐协作流程

### 第 1 步：理解 PRD 与产品边界

将 PRD 交给 AI，要求它先输出：

- 产品目标与成功标准；
- 用户角色及权限边界；
- 核心用户路径；
- 涉及的页面和关键数据；
- 不在本期范围内的能力。

**可直接复制的提示词：**

```text
请阅读以下 PRD，先不要生成页面或代码。

请以产品经理和医疗行业数字化产品设计师的视角输出：
1. 产品要解决的核心问题与本期目标；
2. 用户角色、每个角色可见的功能和权限边界；
3. 一线代表端与管理端的核心用户路径；
4. 全部功能模块、关键数据字段和依赖关系；
5. 不在本期范围内的功能、数据或合规边界；
6. PRD 中存在歧义、缺失或需要我确认的事项。

请用表格和流程顺序表达，不要自行补充医疗诊断、真实 CRM 或 AI 洞察能力。

【附：PRD】
```

**验收标准：** AI 不应直接开始画页面；应先复述产品目的、流程和范围。

### 第 2 步：生成 H5 移动端低保真流程图

在对话式 AI 或设计/开发 Agent 中基于 PRD 生成低保真流程图。提示必须明确：

- 设计基准为 H5 移动端；
- 推荐基准尺寸为 390 × 844；
- 输出页面序号、页面标题、主要字段与跳转箭头；
- 重点是验证产品流程，不追求视觉装饰。

**可直接复制的提示词：**

```text
请基于以下 PRD，生成一张低保真产品流程图。

要求：
- 按 H5 移动端页面设计，基准尺寸按 390 × 844 理解；
- 每个页面标注页面序号、页面名称、主要字段/模块和主要操作；
- 用箭头清楚表达用户跳转；
- 分别展示一线代表端和管理端；
- 必须覆盖新建拜访、创建 1–5 步、竞品卡、访后复盘、行动支持、客户洞察等路径；
- 本图只用于验证流程和信息架构，不需要高保真视觉设计。

生成后，请附一份页面清单，说明每个页面的入口和下一页。

【附：PRD】
```

**目的：** 让团队快速看清“用户从哪里来、做什么、去哪里”，而不是立即确认视觉细节。

### 第 3 步：向设计/开发 Agent 提供完整设计上下文

将以下内容一起给设计/开发 Agent：

- PRD；
- 低保真流程图；
- H5 尺寸要求；
- 视觉参考图；
- 已确认的品牌、颜色、图标或内容限制。

要求 Agent 先总结其理解，并列出不确定项；不要只发送一张参考图要求“做类似风格”。

#### 视觉参考网站

如果团队尚未准备具体截图，可以先在以下网站寻找 3–5 张与目标页面类型接近的参考，再将截图或明确页面链接提供给 AI。

| 类别 | 网站 | 推荐使用方式 |
|---|---|---|
| 移动端与 Web UI | [Mobbin](https://mobbin.com/) | 查找真实 App 的首页、列表、表单、筛选、导航和状态页面。 |
| 移动端与 Web UI | [ScreensDesign](https://screensdesign.com/) | 查找移动产品页面结构、界面密度与导航模式。 |
| 移动端与 Web UI | [Variant](https://variant.com/) | 查找产品 UI、组件组合与视觉方向。 |
| 综合静态设计灵感 | [Pinterest](https://www.pinterest.com/) | 搜索 `app design`、`healthcare app`、`mobile dashboard`、`medical UI` 等关键词，建立风格板。 |
| 综合静态设计灵感 | [Dribbble](https://dribbble.com/) | 搜索卡片、仪表盘、移动端表单、医疗界面等视觉表达；需结合 PRD 判断可用性。 |
| 图标、插画与动效参考 | [LottieFiles](https://lottiefiles.com/) | 查找 Lottie 动效、空状态插画和动态图标灵感；用于开发时须确认授权、文件格式和性能，不应把动效直接当作静态功能图标。 |

**使用原则：** 网站链接本身不是完整视觉输入。应说明每张参考图分别借鉴什么，例如“只参考顶部背景”“只参考筛选抽屉”“只参考图标线条风格”；不要要求 AI 同时融合多个相互冲突的视觉风格。

**可直接复制的提示词：**

```text
你是医疗行业资深数字化产品 UI/UX 设计师。

以下是本工具的设计输入：PRD、H5 低保真流程图、视觉参考图和已确认限制。

请先输出：
1. 你对产品目标、角色、核心路径的理解；
2. 你识别到的页面模式和可复用组件；
3. 视觉参考中应继承的元素，以及不应继承的元素；
4. 需要我确认的产品或内容问题。

约束：设计基准为 390 × 844 H5；不得凭视觉参考推测 PRD 中没有的业务；医疗信息缺失时必须标记“待确认”，不能编造。

在我确认前，请不要生成全量 UI 图。

【附：PRD】
【附：低保真流程图】
【附：视觉参考图】
【附：每张参考图的借鉴说明，例如顶部 / 图标 / 筛选抽屉 / 卡片密度】
```

### 第 4 步：先建立页面资产台账，再设计 2–4 个核心页面

这是避免后续漏页的关键步骤。

#### 4.1 页面资产台账

先要求 AI 根据 PRD 和低保真流程输出完整页面清单。建议使用下表：

| 页面 ID | 角色 | 页面名称 | 入口 | 主要操作 | 去向 | 首批设计 |
|---|---|---|---|---|---|---|
| FL-01 | 一线 | 首页 | Demo 入口 | 新建拜访、客户入口 | 拜访创建 / 客户档案 | 是 |
| FL-02 | 一线 | 拜访列表 | 底部 Tab | 搜索、筛选、查看详情 | 拜访详情 | 是 |
| FL-03~07 | 一线 | 创建拜访 1–5 步 | 新建拜访 | 填写、保存、下一步 | 访前准备 | 选代表页 |
| MG-01 | 管理 | 管理概览 | 管理端入口 | 查看指标、筛选 | 管理明细页 | 是 |
| MG-02 | 管理 | 客户声音 | 底部 Tab | 搜索、筛选、查看详情 | 声音详情 | 是 |

台账应覆盖一线端、管理端、详情页、创建页、二级列表、空状态及关键流程状态。

#### 4.2 核心页面选择

不要随机选择 2–4 张页面。核心页应覆盖不同的设计模式：

1. 一线首页：验证首页信息层级、底部 Tab、快捷入口与“新建拜访”。
2. 一线创建拜访代表页：验证表单、五步进度、底部固定操作区。
3. 管理概览：验证管理端导航、指标、时间和组织范围筛选。
4. 管理列表页：验证局部搜索、筛选抽屉、状态标签、列表密度和脱敏信息。

**验收标准：** 确认一个统一视觉方向，而不是让不同参考图分别演变成互相矛盾的 UI 风格。

**可直接复制的提示词：**

```text
请基于 PRD、低保真流程图和视觉参考，先不要补齐所有 UI 图。

第一步，请输出《页面资产台账》：包含页面 ID、用户角色、页面名称、入口、主要操作、下一页、默认状态、关键特殊状态，以及是否属于首批核心页面。

第二步，请输出完整用户路径，分别覆盖一线代表端与管理端；重点检查客户入口、客户详情、创建拜访 1–5 步、竞品卡、访后复盘、行动支持、管理拜访列表和管理客户洞察是否遗漏。

第三步，请从不同页面模式中选择 2–4 个核心页面生成 H5 UI 图：一线首页、一线创建拜访代表页、管理概览、管理列表页。基准尺寸为 390 × 844。

要求：所有核心页使用同一视觉方向；每张卡片最多使用一个主 SVG 图标；请同时给出每张核心页面对应验证的全局组件和交互规则。

【附：PRD】
【附：低保真流程图】
【附：视觉参考图】
```

### 第 5 步：确认全局设计与交互规则

在生成全部页面之前，先形成一份全局规则表。至少确认：

- 一线端、管理端各自的一级 Tab 名称、顺序和激活态；
- 哪些页面显示底部 Tab，哪些二级页必须隐藏；
- 搜索在哪些列表页出现；
- 搜索支持哪些字段；
- 筛选抽屉结构、默认时间范围与已选条件反馈；
- 创建拜访 1–5 步的顶部、进度条、底部操作区；
- 客户快捷入口、客户详情、新建拜访之间的跳转；
- 顶部背景、图标、卡片、按钮、标签、状态色和留白规则。

对于 Call Insight，已经确认的示例规则：

| 项目 | 规则 |
|---|---|
| 一线端 Tab | 首页 / 拜访 / 洞察 / 我的 |
| 管理端 Tab | 概览 / 客户声音 / 行动支持 / 我的 |
| 客户 | 首页快捷入口，不作为一线一级 Tab |
| 搜索 | 仅列表页；支持医院、科室、客户姓名、负责人姓名 |
| 筛选 | 顶部保留时间范围和筛选入口；完整条件使用底部抽屉 |
| 创建流程 | 统一为 1/5 至 5/5 顶部、五步进度条和底部固定操作区 |
| 顶部视觉 | 深蓝实底、几何弧线与节点、蓝白实线分区；不使用蓝白大渐变 |
| 图标 | 使用统一 SVG 图标库；每张卡片保留一个主图标 |

**可直接复制的提示词：**

```text
请基于已确认的核心 UI 页面，先不要生成更多页面。

请输出《全局 UI 与交互规则表》，逐项明确：
1. 一线端和管理端的一级 Tab：名称、顺序、激活态、适用页面；
2. 哪些页面显示底部 Tab，哪些二级页必须隐藏；
3. 搜索出现在哪些页面，统一搜索字段和占位文案；
4. 每类页面的筛选项、默认值、底部筛选抽屉与已选条件反馈；
5. 创建拜访 1–5 步的统一顶部、进度条、步骤名称、固定底部操作区；
6. 客户入口、客户详情、新建拜访、拜访详情、复盘和行动之间的跳转；
7. 顶部、图标、卡片、按钮、标签、状态色、留白的视觉规则。

请先列出“逐页影响清单”，不要直接批量改图。对未确认规则标注“待确认”。

【附：已确认核心 UI 图】
【附：页面资产台账】
```

### 第 6 步：确认页面内容与关键状态

在全量出图前，逐页补充下列内容：

- 页面目的；
- 入口与主要操作；
- 信息分组与字段层级；
- 默认展示状态及示例数据；
- 空数据、待办、已完成、报错、不可编辑等必要状态；
- 页面之间的跳转关系。

优先确认关键任务页和高风险页面；普通展示页可在保持信息层级不变的前提下后补文案。

**可直接复制的提示词：**

```text
请以《页面资产台账》和《全局 UI 与交互规则表》为依据，为每个页面输出《页面内容确认表》。

每页必须包含：
- 页面 ID、角色、页面目的；
- 入口和主要操作；
- 信息分组、字段层级与必填项；
- 一个默认展示状态的示例数据；
- 必要特殊状态：空数据、待办、已完成、报错、不可编辑；
- 页面出口和跳转条件。

请优先列出影响主流程和开发实现的内容缺口；对于医疗或客户信息，不可编造，统一标记为“待确认”。

【附：页面资产台账】
【附：全局 UI 与交互规则表】
```

### 第 7 步：基于页面资产台账补齐全部 UI 图

此时 AI 的任务应以页面资产台账为唯一依据，逐项核查和补齐。

全量页面不是一次性无序生成，建议按批次进行：

1. 全局组件基准；
2. 一线端一级页面；
3. 拜访详情与创建 1–5 步；
4. 访后复盘与行动闭环；
5. 客户档案与客户详情；
6. 管理端一级页面；
7. 管理端明细页与洞察页。

每完成一个批次，确认导航、搜索、筛选、状态和页面跳转，再进入下一批。

**可直接复制的提示词：**

```text
请以《页面资产台账》《全局 UI 与交互规则表》《页面内容确认表》为唯一依据，分批补齐所有 H5 UI 图。

请先输出本批次的页面 ID、页面名称、入口、出口、复用组件和修改点；待我确认后再生成 UI 图。

补图要求：
- 不得遗漏台账中的任何页面 ID；
- 不得仅根据已生成页面推测功能；
- 一线端与管理端分别遵守各自固定 Tab；
- 搜索、筛选、顶部和底部操作区必须遵守全局规则；
- 每页按 390 × 844 输出；
- 每张卡片最多一个主 SVG 图标；
- 对新增页面标注其入口、出口和复用组件。

本次只处理以下批次：
【填写本批次页面 ID，例如：FL-01、FL-02、FL-08】

【附：三份确认表】
```

### 第 8 步：全局验收与流程图复核

将完成的页面重新排成完整用户路径图，逐项检查：

- 每个页面 ID 是否都有 UI 图；
- 一线端与管理端的 Tab 是否统一；
- 二级页是否错误显示底部 Tab；
- 搜索是否只出现在规定的列表页；
- 同类筛选是否采用相同抽屉与条件反馈；
- 创建拜访 1–5 步是否完全统一；
- 客户入口、竞品卡、复盘、行动支持、管理明细等二级路径是否遗漏；
- H5 页面是否在 390 × 844 下无横向裁切，底部操作区可见。

**可直接复制的提示词：**

```text
请将已完成的全部 UI 页面按真实用户使用路径重新排成一张流程图，并基于《页面资产台账》执行全局验收。

请输出：
1. 完整用户路径图，区分一线代表端与管理端；
2. 页面覆盖矩阵：每个页面 ID 是否已有 UI 图、默认状态和特殊状态；
3. 导航、搜索、筛选、创建拜访 1–5 步、客户入口、竞品卡、复盘、行动和管理明细页的一致性检查；
4. 缺失页面、跳转断点、规则冲突和 H5 裁切风险清单。

先只输出验收报告与修改清单，不要直接重绘页面。

【附：页面资产台账】
【附：全部 UI 图】
```

### 第 9 步：生成开发交付包

所有 UI 图和流程确认后，再输出开发需要的资料：

1. 设计规范：颜色、字号、字重、间距、圆角、阴影、边框、栅格、安全区。
2. 组件清单：顶部、底部 Tab、搜索框、筛选抽屉、按钮、卡片、列表项、标签、表单项、空状态、弹窗等。
3. 设计 Token：可直接映射到 CSS、Tailwind、React Native 或小程序变量。
4. SVG 图标资源：图标名称、尺寸、颜色规则、使用位置与源文件。
5. 页面说明：页面 ID、字段、状态、跳转、权限与默认示例数据。
6. 截图验收标准：以 390 × 844 为基准，供开发完成后与设计图对照。

**可直接复制的提示词：**

```text
所有页面 UI 图和用户路径已经确认。请生成《开发交付包》，供前端和产品开发使用。

交付包必须包含：
1. 设计规范：颜色、字体、字号、字重、间距、圆角、边框、阴影、栅格、安全区；
2. 组件清单：组件名称、用途、状态、尺寸、适用页面；
3. 设计 Token：以可直接映射至 CSS 变量的形式输出；
4. SVG 图标资产清单：图标名称、尺寸、颜色、使用位置、来源；
5. 页面说明：页面 ID、字段、默认数据、特殊状态、跳转、权限；
6. 截图验收标准：390 × 844 下的比对要点。
```
