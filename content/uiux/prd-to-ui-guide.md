# Part A｜从 PRD 迈向 UI：把想法做成可交付产品

## 这一课要完成什么

你会把一个模糊的业务想法，推进为一套可以交给设计 Agent、前端 Agent 和业务方复核的 UI 交付包。练习项目是 **Call Insight**：帮助一线人员在客户拜访前准备、拜访后复盘，并让管理者看到聚合洞察。

## 先补齐 8 类信息

开始画页面前，先把以下信息写清楚。缺少其中任何一项，后续生成的 UI 都容易“好看但不能用”。

- 定位与背景：产品为什么存在，当前痛点是什么。
- 目标与非目标：本次要解决什么，明确不做什么。
- 角色与权限：一线人员、管理者分别能看什么、改什么。
- 流程与状态：一次任务从发起到完成经过哪些步骤。
- 范围与优先级：MVP 必做、可选和暂不做内容。
- 页面与功能规则：每个页面的主任务、输入、输出与跳转。
- 数据、指标与异常：数字来源、空状态、保存失败和权限不足如何显示。
- 边界与验收：合规红线、不能由 AI 推断的内容、完成的判断标准。

## 第 1 步：先理解 PRD，不急着画图

把 PRD 中的目标用户、核心任务、业务流程和异常情况提取成一页“设计输入”。此时最重要的问题不是“页面长什么样”，而是“用户打开后必须先完成什么”。

对于 Call Insight，一线人员的主链路是：新建拜访 → 填写客户与目标 → 选择产品和竞品 → 查看访前准备 → 访后复盘 → 生成行动项。管理者的辅助链路是：查看概览 → 查看客户声音 → 跟进支持需求。

## 第 2 步：生成低保真流程图

低保真图只解决信息层级、页面关系和操作顺序，不解决配色与装饰。每个框都要能回答三个问题：用户在这里做什么？完成后去哪？出错时怎么办？

推荐先画最短闭环，再补辅助路径。Call Insight 的低保真图把新建拜访拆成 5 步，令一线主路径和管理视角既有关联又不混淆。

## 第 3 步：建立页面资产台账

把页面逐一列入台账，避免设计 Agent 只画出“几个好看的页面”却漏掉关键状态。台账至少包含：页面编号、页面名称、服务角色、主要任务、关键输入、关键输出、前置页面、下一步页面，以及空/错/加载状态。

先挑 2–4 张页面确定风格：入口页、首页、五步向导中的一页、管理概览页通常足够。确认后，再按台账补齐其余页面。

## 第 4 步：写清全局设计与交互规则

- 为不同角色设置清晰的导航和权限边界。
- 指定字号层级、间距单位、圆角、卡片阴影和强调色。
- 明确主按钮、次按钮、禁用、完成、风险和空状态的表达方式。
- 约定表单校验、保存反馈、返回行为和长内容展开规则。

这样做的目的不是限制创意，而是让 AI 在后续补图时稳定执行同一套规则。

## 第 5 步：全局验收与开发交付

逐条对照页面资产台账、业务流程和使用边界。重点检查：主路径是否能从头走到尾；一线与管理权限是否串台；数据来源是否明确；错误和空状态是否有回应；开发所需的组件、状态和文案是否齐全。

最终交付包应包含：PRD、低保真流程图、页面资产台账、核心高保真页面、全局设计规则、完整 UI 图、DESIGN.md，以及开发验收清单。

## 练习清单

1. 用你的项目写出 8 类设计输入。
2. 画出一条最短用户闭环，并标出至少 3 个异常状态。
3. 建立页面资产台账，选出第一批 2–4 张核心页面。
4. 为 Agent 写一份全局视觉与交互规则。
5. 用验收清单复核后，才进入高保真与开发阶段。
