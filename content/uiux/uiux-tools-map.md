# Part C｜UI/UX 工具地图：在正确的时刻使用正确的工具

## 这一课要解决什么

同一份业务需求，可能需要不同类型的工具。关键不是“哪个工具最强”，而是先判断你现在卡在哪个阶段：没有视觉方案、已有页面但不可交付、流程已通但操作不顺，还是风格已定但 AI 总是跑偏。

## 场景 1：只有需求，还没有第一版界面

当 PRD 说明了功能，却没有定义页面该怎样表达时，先使用 **UI/UX Pro Max** 一类的生成工具探索结构、版式和视觉方向。输入中应包含用户、任务、页面清单、内容优先级和参考风格；不要让它替你猜业务规则。

产出：2–4 张核心页面的视觉方向，以及可继续讨论的第一稿。

## 场景 2：页面能跑，但离交付还有距离

当你已经有可用页面，却发现层级混乱、间距不稳、组件不一致或细节粗糙时，使用 **Impeccable** 一类的审阅与优化工具。它的任务不是重新设计一张更炫的图，而是逐项找出可交付性问题并提出可验证的修改。

产出：更一致的组件规则、布局修正清单和视觉质量提升。

## 场景 3：结构没问题，但用起来不够顺

当页面流程成立，却缺少清晰的反馈、加载、过渡和操作节奏时，使用 **Make Interfaces Feel Better** 一类的交互打磨方法。关注用户每次操作后“看到了什么、知道了什么、下一步做什么”，而不是为动效而动效。

产出：更明确的点击反馈、状态转场、错误提醒和微交互规范。

## 场景 4：风格已选好，怎样让 AI 稳定执行

当参考风格只停留在图片上，Agent 每次都会凭感觉模仿。此时用 **Awesome DESIGN.md** 一类的设计规范方法，把颜色、字体、空间、组件、图标、状态和禁止事项写成可执行规则。

产出：能被设计 Agent 和开发 Agent 共同读取的 DESIGN.md，后续改一页不必重做全站。

## 最实用的选择顺序

1. 先用生成工具探索第一稿。
2. 用页面审阅工具把第一稿修到可交付。
3. 用交互方法补足操作反馈和状态。
4. 把最终规则沉淀进 DESIGN.md，再交给开发 Agent。

工具只是放大器。业务流程、信息层级和验收标准仍然必须来自你的 PRD 与线框图。
