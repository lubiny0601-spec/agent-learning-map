UI/UX问题 01

只有需求，页面该怎么设计？

01
Problem First

需求说明了功能，但还没有定义界面该怎么表达。

如果不先确定用户场景、产品气质、信息层级和组件规则，Agent 往往只能生成一次性页面：单页能看，多页难统一。

目标用户不明视觉规则缺位多页表达发散
Recommended Skill

UI/UX Pro Max

GitHub ↗

先分析产品类型、用户和场景，再生成设计方向与设计系统。

README 关键点先产出设计系统：风格、色彩、字体、组件、状态、反模式和响应式规则。
适合场景需求已清楚但缺少 UI 方向，尤其适合多页面产品从 0 到 1。
General Prompt用于“从需求到设计系统”
请使用 UI/UX Pro Max。

我会提供【产品 / 页面需求】。请先不要直接画页面，先判断：目标用户是谁、核心任务是什么、主要使用场景和设备是什么、这个产品应该给人什么感觉。

然后先产出一套可执行的 UI 方向：视觉风格、信息层级、颜色角色、字体层级、间距、组件规则、页面状态、响应式规则，以及明确不要采用的设计做法。

最后再基于这套方向完成页面。保留原业务内容和功能，不新增未确认模块。完成后检查多页面是否统一、移动端是否可用、文字和按钮状态是否清楚。
UI/UX问题 02

页面能跑，但用户不好完成任务？

02
Problem First

不是再做一版更好看的页面，而是把现有页面改到可交付。

Impeccable 会像资深产品设计审查一样，从真实任务出发检查层级、流程、状态、错误恢复和移动端体验，然后按优先级直接改。

交付前审查任务路径直接改造
Recommended Skill

Impeccable

GitHub ↗

把已有页面当成真实产品来审查：先找阻碍用户完成任务的问题，再直接改到更可交付。

README 关键点建立上下文后，用 critique / audit / harden / polish 做产品级审查、修复和收尾。
适合场景已有页面雏形，但流程、状态、表单、响应式或交付质量还不够可靠。
General Prompt用于“已有页面的系统审查与改造”
请使用 Impeccable。

请把当前【页面 / 产品】当成一个准备交付的真实产品来审查，而不是只看视觉好不好看。重点检查：用户的主任务是否一眼明确、信息顺序是否支持完成任务、下一步操作是否清楚、表单和按钮状态是否完整、错误 / 成功 / 空状态是否能帮助用户继续、移动端是否真的可点可读。

请按严重程度直接修改页面。可以调整布局、模块顺序、信息层级和状态呈现，但保留原业务内容、功能和数据，不新增未确认模块。完成后说明：你修掉了哪些会阻碍用户完成任务的问题。
UI/UX问题 03

结构没问题，但操作总差一点？

03
Problem First

流程已经成立，但界面反馈还没有形成顺手的操作感。

问题通常藏在对齐、点击区域、数字稳定、hover / pressed / focus、加载和过渡里；它们不改业务，却决定质感。

触达不稳状态反馈弱微交互粗糙
Recommended Skill

Make Interfaces Feel Better

GitHub ↗

保持结构不动，专门打磨用户能感受到、但很难逐项写进需求的界面细节。

README 关键点聚焦排版、表面层级、动效、性能，以及 hover / pressed / focus 等交互状态。
适合场景结构和流程已确认，准备演示或交付，需要不推翻设计的最后一轮精修。
General Prompt用于“结构不动的最后一轮精修”
请使用 Make Interfaces Feel Better。

请在不改变【页面】业务内容、信息结构、主要布局和用户流程的前提下，做最后一轮界面手感精修。

重点检查这些细节：文字层级和换行是否舒服，数字变化是否导致布局跳动，卡片 / 按钮 / 输入框是否对齐，父子圆角是否协调，hover / pressed / focus / disabled / loading / error / success 状态是否完整，点击区域是否足够大，动画是否轻、快、不会打扰用户。

不要重做页面，不新增业务功能。完成后列出最明显提升“操作感”的修改。
UI/UX问题 04

风格已选好，怎样让 AI 稳定执行？

04
Problem First

参考风格如果没有规则化，Agent 只能凭感觉模仿。

需要把颜色、字体、组件、间距、表面层级和禁用做法写成可执行约束，才能跨页面稳定复用。

只学表面执行漂移规则不可复用
Recommended Resource

Awesome DESIGN.md

GitHub ↗

它不是 Skill，而是可放进项目的 DESIGN.md 资源：让 Agent 按选定视觉语言重建页面。

怎么用选定 DESIGN.md，放到项目根目录，并要求 Agent 完整读取后再开始重建 UI。
适合场景已有明确视觉参考，希望多个页面长期遵守同一套颜色、字体、组件和状态规则。
General Prompt用于“已有明确风格的整套视觉重建”
请先完整读取项目根目录中的 DESIGN.md。

请保留当前【页面 / 产品】的业务内容、功能、流程和数据，但用 DESIGN.md 中的规则重新建立视觉表达。重点执行：颜色角色、字体层级、间距和栅格、按钮、卡片、表单、导航、表面层级、圆角、边框、阴影、交互状态和响应式规则。

不要只换主色或背景色。没有被明确写到的组件，请根据 DESIGN.md 的整体原则推导；不要回退到旧样式。不要复制参考品牌的名称、Logo、原始文案或专属素材。完成后逐页检查一致性，并列出你实际应用了哪些设计规则。
